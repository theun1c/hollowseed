# Подход разработки, где 3 отдельных чата 
### Что писать в новый чат (шаблон)

Скопируй и используй почти без изменений:

Работаем только по одной фиче.

Прочитай файлы:
1) docs/codex/CONTEXT.md
2) docs/codex/features/feat-00X/SPEC.md
3) docs/codex/features/feat-00X/TASKS.md
4) AGENTS.md

Ограничения:
- только scope этой фичи
- TDD-first (RED -> GREEN -> REFACTOR)
- минимальные изменения
- не трогать несвязанные файлы

Текущий шаг:
<опиши 1 конкретный шаг, например: "написать первый RED-тест для
NewGrid">

Формат ответа:
1) кратко подход
2) минимальный патч
3) как проверить
4) что дальше

### Как держать единый контекст между разными моделями

1. Один источник истины: CONTEXT.md + SPEC.md + TASKS.md.
2. Каждый чат начинает с ссылки на те же 3 файла.
3. После каждого шага обновляй TASKS.md и при необходимости SPEC.md
   (decision log).
4. Следующий чат стартует с фразы:
   Продолжай с текущего состояния TASKS.md, не переопределяй принятые
   решения.

### Про “разные модели на одну фичу”

Это нормально. Практичный режим:

1. high — дизайн/архитектурные решения и декомпозиция.
2. medium — основная реализация по TDD.
3. low — локальный дебаг, мелкие правки, cleanup.

Главное: модель меняется, но входной контекст всегда один и тот же (те
же файлы + тот же branch feat/...).

### Минимальная дисциплина на фичу

1. git switch -c feat/<name>
2. обновить SPEC/TASKS
3. чат A (high): уточнить дизайн шага
4. чат B (medium): реализовать шаг TDD
5. чат C (low): дебаг/фиксы
6. коммит с ссылкой на feat-00X

# Flow на 3 чата 

Ниже универсальный flow для 1 фичи в 3 чатах с общим контекстом.

## 1) Единый контур фичи (до чатов)

Для каждой фичи держи:

- docs/codex/features/feat-XXX/SPEC.md
- docs/codex/features/feat-XXX/TASKS.md
- docs/codex/features/feat-XXX/DECISIONS.md (короткий лог решений)

И всегда общий:

- docs/codex/CONTEXT.md
- AGENTS.md

## 2) Роли чатов

1. Chat A (high) — дизайн/декомпозиция
   Результат: обновлённые SPEC/TASKS/DECISIONS.
2. Chat B (medium) — реализация TDD
   Результат: код + тесты + отметки в TASKS.
3. Chat C (low) — дебаг/чистка/локальные фиксы
   Результат: маленькие исправления без смены дизайна.

## 3) Шаблоны сообщений (универсальные)

### Шаблон старта любого чата

Работаем в рамках одной фичи.

Прочитай:
- AGENTS.md
- docs/codex/CONTEXT.md
- docs/codex/features/feat-XXX/SPEC.md
- docs/codex/features/feat-XXX/TASKS.md
- docs/codex/features/feat-XXX/DECISIONS.md

Ограничения:
- только scope фичи
- минимальные изменения
- TDD-first (если есть код-изменения)
- не трогать несвязанные файлы

Перед изменениями: кратко зафиксируй, что понял из SPEC/TASKS.

### Для Chat A (high)

Задача: уточнить дизайн и разбить фичу на шаги.
Выход:
1) 2-3 подхода с trade-offs
2) выбранный подход
3) обновить TASKS/DECISIONS
Код не писать.

### Для Chat B (medium)

Задача: реализовать только шаг <N> из TASKS через TDD.
Выход:
1) RED тест
2) минимальный GREEN код
3) REFACTOR (если нужно)
4) что отмечено в TASKS

### Для Chat C (low)

Задача: исправить конкретный дефект/нестабильность в шаге <N>.
Не менять архитектуру и scope.
Выход: минимальный фикс + проверка + запись в DECISIONS.

## 4) Flow по шагам

1. Создаёшь ветку: feat/XXX-name.
2. Chat A (high) → уточняет дизайн, правит SPEC/TASKS/DECISIONS.
3. Chat B (medium) → делает шаг 1 TDD, коммит.
4. Chat C (low) → чинит локальные проблемы (если есть), коммит.
5. Повторяешь цикл для следующего шага TASKS.
6. В конце: сверка с Acceptance Criteria в SPEC.

## 5) Как не терять контекст между чатами

- Каждый чат стартует с одних и тех же файлов.
- Истина только в документах, не в “памяти чата”.
- После каждого заметного решения обновляй DECISIONS.md.
- После каждого шага обновляй чекбоксы TASKS.md.
